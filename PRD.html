제안서 1건을 메시지 · 논리구조 · 내용 3-tuple 로 분해 학습.

UD Labs 의 6 product (Coach Finder · Coaching Log · ActionAI · AILaunchpad · UCA Textbook · ud-ai-guidelines) 가 자산 source. Sphere 2 (AI 두뇌) 는 제안서를 단어 chunk 가 아닌 의미 단위 3-tuple 로 학습 — 메시지 (의미 vector + 톤 패턴) · 논리구조 (Logic Model graph) · 내용 (MMR 다양성 강제 RAG).

"제안서 하나를 학습하더라도 전체 맥락들과 메세지 그 안에서 제안서 논리구조와 내용들 이런 것들도 학습을 하는 Sphere 2 가 되어야해. 잘못 RAG 청크하면 다 단어 단위만 남고 또 막 특정 단어들이 과하게 노출되거나 계속 동일한 단어가 반복해서 출력되거나 하니까." — udpb@udimpact.ai · 2026-05-23 (v11.0 트리거)

Section 00 · Executive Summary

요약 — 5 본질

UD-Planner 의 청사진 한 페이지

  1. 심지 — 언더독스의 에셋 + RFP + 학습된 자산 → 제안서 자동화. PRD 의 모든 결정이 이 한 줄의 표현. 자료가 들어와도 심지는 바뀌지 않음
  2. UD-Planner = P1 + hub — 11 product 중 P1 이지만, P2 가이드북 · P5 Coach Finder · P6 Coaching Log · P11 KSVA 와의 통합 청사진. 근거: ud-ai-guidelines/UD-products-PRD-v2.html §P10 통합 매핑
  3. Sphere 2 = 3-tuple 학습 — 제안서 1건 ingest 시 LLM 3 호출로 메시지 · 논리구조 · 내용 분해. 매칭 시 3 차원 병렬. 단어 chunk 만의 RAG 한계 대응. Chapter 4 가 PRD 의 핵심
  4. 데이터 허브 (P10) = Sphere 2 학습 source — 7 자산 (사업 마스터 · 학습 진도 · 코칭 일지 · 예산 · 임팩트 · 제안서 아카이브 · 코치 평가). xAPI 표준 Q2 통합 청사진. 3-tuple 의 자연 분해와 정합
  5. 채널별 가중치 + tonePatterns — B2G/B2B/renewal 별 매칭 가중치 + 수주 사례의 표현 패턴 (openings · transitions · avoidedWords) 학습. 반복 출력 방지 메커니즘

"가벼운 PM 도구 위에, 자산이 자라면 제안서가 자란다. 단, 그 학습이 단어가 아니라 메시지·논리·내용의 3-tuple 위에서 이뤄진다."

Section 4.1 · 3-Tuple Definition

3-tuple 정의

제안서 1건 = (Message + LogicStructure + Content)

RAG 의 4 한계 (단어 매몰·과노출·반복·맥락 손실) 는 모두 "제안서를 단어 단위 chunk 로만 학습한다" 는 단일 원인. 해소책은 제안서를 의미 단위 3 차원으로 분해.

1️⃣
MESSAGE

메시지 — 설득의 핵심

제안의 정체성 · "왜 우리인가"
핵심 1줄 슬로건 + 키 메시지 3개 + Before/After 차이 + 톤 패턴. 단어 X — 의미 vector 와 표현 패턴.

예: "데스밸리 통과 95%" · "실행 보장 시스템" · tone: openings=["우리는 ··· 합니다"]
DB WinningPattern.message
WinningPattern.tonePatterns 매칭 cosine + 채널 가중치 L1 입력 S2 "메시지 결정" Stage
2️⃣
LOGIC

논리구조 — 섹션 간 인과 chain

"무엇 → 무엇" graph · 맥락의 chain
Activity → Output → Outcome → Impact (Logic Model). 섹션 간 인과 관계 graph (nodes + edges). chunk X — 전체 chain 보존.

예: S1 시장 통계 → S2 문제 정의 → S3 해법(우리 자산) → S6 KPI 순서·의존성 모두 graph 로
DB WinningPattern.logicGraph
(nodes + edges JSON) 매칭 subgraph isomorphism + edit distance L1 입력 S3 "논리구조" Stage
3️⃣
CONTENT

내용 — 구체 fact · 수치 · 인용

"숫자 · 출처 · 인용" · 평가위원 검증 대상
각 섹션의 fact · 수치 · 인용 (RAG chunk). hybrid retrieval (BM25 + embedding) + MMR 다양성 강제.

예: "누적 20,211명" · "운영 32개월" · "[근거: 통계청 2026.03]"
DB ContentAsset.embedding
ContentAsset.keyNumbers 매칭 BM25 + embedding + MMR L1 입력 S4 "내용 작성" Stage

핵심 통찰: 제안서 1건을 단어 단위 chunk 로만 학습하면 RAG 4 한계 모두 발생. 3-tuple 분해 시:

• Message 는 단어 X — 의미 vector → 단어 매몰 해소

• LogicStructure 는 graph — 섹션 간 chain → 맥락 손실 해소

• Content 는 MMR 다양성 → 과노출 + 반복 해소

Section 4.4 · Matching Algorithm

매칭 알고리즘 — 실제 구현

새 RFP 의 3-tuple 후보 → 3 차원 병렬 매칭 → 통합

새 RFP 가 들어오면 Sphere 2 가 3 차원 각각에서 매칭. 결과를 통합해 LLM prompt 에 주입.

매칭 점수 — 통합 공식
// 한 WinningPattern + ContentAsset 후보의 매칭 점수 (0~1)
matchScore =
  0.35 × messageSim // cosine(rfpMessageVec, patternMessageVec)
+ 0.30 × logicSim // graph 유사도 (subgraph + edit distance)
+ 0.20 × contentSim // BM25 + embedding hybrid
+ 0.10 × channelMatch // 채널 일치 가중치
+ 0.05 × winRateBonus // 과거 win-rate (Laplace + half-life 1yr)

// MMR 다양성 강제 (top N 결과)
mmrScore = 0.7 × matchScore − 0.3 × maxSimToSelected

Step 23 차원 병렬 매칭:
· Message: pgvector cosine query top 20
· LogicGraph: graph similarity top 20
· Content: hybrid (BM25 + embedding) top 50

매칭 pseudocode
함수 matchTuple(rfp, profile, channel, limit=10):

  // 1. RFP 의 3-tuple 후보 추정 (LLM)
  rfpTuple = extractRfpTuple(rfp, profile, channel)
  // rfpTuple = {messageVec, logicGraph, contentKeywords, contentEmbedding}

  // 2. 3 차원 병렬 매칭
  [messages, logics, contents] = Promise.all([

    // Message 매칭 — pgvector cosine
    prisma.$queryRaw`
      SELECT id, 1 - (messageVector <#> ${rfpTuple.messageVec}) AS sim,
             channel, outcome, message
      FROM winning_pattern
      WHERE outcome IN ('won', 'pending')
      ORDER BY messageVector <#> ${rfpTuple.messageVec}
      LIMIT 20
    `,

    // LogicGraph 매칭 — graph2vec 기반 cosine + edit distance 결합
    matchLogicGraphs(rfpTuple.logicGraph, {
      topK: 20,
      method: 'hybrid'  // graph2vec cosine 0.7 + edit distance 0.3
    }),

    // Content 매칭 — BM25 + embedding hybrid
    hybridContentSearch({
      bm25Query: rfpTuple.contentKeywords,
      embedding: rfpTuple.contentEmbedding,
      filter: { channel, sourceTier: ['high', 'internal'] },
      topK: 50
    })
  ])

  // 3. 통합 점수 + 채널·win-rate 가중치
  scored = []
  for m in messages:
    pattern = prisma.winningPattern.findUnique({ id: m.id })
    channelMatch = (pattern.channel === channel) ? 1.0 : 0.5
    winRate = computeWinRate(pattern.sourceProject)  // Laplace + half-life

    matchScore =
      0.35 * m.sim
      + 0.30 * findInArray(logics, m.id)?.score ?? 0
      + 0.20 * avgContentSim(pattern.contentRefs, contents)
      + 0.10 * channelMatch
      + 0.05 * winRate

    scored.push({ patternId: m.id, matchScore, message: pattern.message })

  scored.sort((a, b) => b.matchScore - a.matchScore)

  // 4. MMR 다양성 강제 (반복 출력 방지)
  selected = []
  for candidate in scored:
    if selected.length >= limit: break

    maxSim = max(selected.map(s => cosine(candidate.messageVec, s.messageVec)))
    mmrScore = 0.7 * candidate.matchScore - 0.3 * maxSim

    if mmrScore >= 0.45:  // threshold
      selected.push(candidate)

  return { messages: selected, logics, contents }

매칭의 본질: 3 차원 (메시지·논리구조·내용) 병렬 매칭 + 통합 점수 + MMR 다양성. 단어 단위 cosine 한 번이 아닌, 의미 단위로 3 번 매칭 후 통합 점수 결합.

Section 4.5 · Generation Rules

생성 시 강제 룰

LLM prompt 에 다양성 + 톤 일관성 강제

매칭 결과를 LLM 에 그냥 주입하면 — LLM 이 표면 단어를 복사해 같은 표현 반복. 강제 룰 4 가지:

  1. tonePatterns.avoidedWords 명시 — prompt 에 "다음 단어 사용 금지: [...avoidedWords]" 직접 명시. LLM 이 그 단어 회피. 반복 단어 방지
  2. "이전 섹션과 다른 표현으로" 강제 — 다음 섹션 생성 시 prompt 에 이전 섹션 텍스트 일부 포함하고 "이전과 다른 어휘로" 명시. 반복 출력 방지
  3. MMR 결과 다양성 ≥ 0.7 — 매칭 단계에서 이미 적용 → LLM 받는 자산이 표현 다양. 과노출 방지
  4. logicGraph 의 sectionOrder 강제 — sections 순서는 매칭된 logic chain 따름. PM 이 임의 변경 X. 맥락 손실 방지
