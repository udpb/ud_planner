# ADR-019: 과업(Workstream) 레이어 — 제안서를 "교육 1종"에서 "N과업 합성"으로

- **상태**: **Accepted** (사용자 승인 "이대로 가자", 2026-06-01)
> taxonomy 1차 검증(2026-06-01): eval RFP 픽스처 6종(청년창업 발굴·육성 등)이 모객·심사_선발·교육·멘토링_코칭·교류_네트워킹 과업으로 깔끔히 분해됨 → 6종+확장(교육·행사_운영·특강연사·장소)이 도메인 커버 확인. **전체 148 당선본 대상 검증은 WS-1 브리프 후속.**
- **결정일**: 2026-06-01
- **결정자**: udpb@udimpact.ai + AI Architect
- **관련**: ADR-006 (ProgramProfile 11축 — 과업유형 6종의 출처), ADR-008 (Impact Value Chain), ADR-011 (Express 메인), ADR-013 (Express 2.0 자동 진단), ADR-018 (Adaptive Stage Layout)
- **Scope**: 데이터 모델 + Express 생성 스키마 + 학습 루프(ingest·RAG) + 출력(7섹션·키메시지) 전체

---

## 배경 (Context)

### 문제 — 그릇이 사업의 실제 모양과 다르다

언더독스 제안서는 **보통 여러 과업이 모여 하나의 프로젝트**가 된다. 예: `청년창업 교육(교육) + 데모데이(행사_운영) + 특강 연사 3명(교류·교육) + 장소 섭외(자원) + 홍보·모객(모객)`. 키메시지는 이 과업들을 가로질러 서고, 각 키메시지는 **과업 단위의 디테일한 근거**로 뒷받침되어야 당선된다.

그런데 현재 시스템의 그릇(`src/lib/express/schema.ts`)은 **교육 사업 1종에 하드코딩**돼 있다:

```
7섹션 고정: ①제안배경 ②추진전략 ③교육 커리큘럼 ④운영체계 및 코치진 ⑤예산 ⑥기대성과 ⑦수행실적
슬롯/UI: CurriculumBoard · 코치풀 · 회차수 — 전부 교육 모양
keyMessages: 최대 3개 (string[]), messageHierarchy(3-3)는 있으나 과업별 조직 없음
```

→ `행사_운영`·`특강연사`·`장소`는 **데이터 모델에 존재하지 않는다.** Section ③이 곧 "교육 커리큘럼"이라 과업이 여러 개인 프로젝트를 담을 자리가 없다.

### 결정적 정렬 깨짐 — 학습 투자가 출력에 안 닿는다

2024-2025 당선 제안서 148건을 풀텍스트 학습(WinningProposalDoc/Chunk 2048)시켰다. 그 안에는 행사운영·연사 구조와 그걸 뒷받침하는 당선 언어가 들어있다. 그러나 **생성 스키마가 그 구조를 담을 그릇이 없어서**, RAG가 당선 언어를 검색해 와도 구조적으로 배치할 자리가 없다. **학습한 만큼 출력이 좋아지지 않는 근본 원인이 여기 있다.**

### 이미 있는 씨앗 — ProgramProfile 과업유형 6종 (ADR-006)

다행히 과업 개념의 씨앗은 이미 있다. `src/lib/program-profile.ts`의 축 9 methodology = **과업유형 6종**이며, 제1원칙으로 **각 유형이 RFP 평가 배점 한 카테고리에 직접 연결**된다:

| 과업유형 | 연결 배점 |
|---|---|
| 모객 | 모집 전략 |
| 심사_선발 | 심사·선정 설계 |
| 교류_네트워킹 | 차별화(파트너·동문 자산) |
| 멘토링_코칭 | 수행역량(4중 지원) |
| 컨설팅_산출물 | 수행능력(산출물 수준) |
| **행사_운영** | **운영역량·집객 실적** |

문제는 이 6종이 **사업을 *서술*하는 프로파일 축**으로만 소비되고, 제안서를 **구성하는 골격**으로는 안 쓰인다는 것. 이 ADR은 그것을 골격으로 승격한다.

---

## Options Considered

### Option A — 7섹션 고정 유지 + Section ③을 "사업 내용"으로 일반화
- 장점: 변경 최소. UI/렌더 대부분 보존.
- 단점: 과업 간 예산·근거·키메시지 연결이 여전히 평문 안에 묻힘. 학습 구조 정렬 미해결. "과업별 디테일 근거"가 구조화 안 됨.
- 기각: 증상만 가림. 사장님 목표(과업별 키메시지·근거)를 구조로 못 받음.

### Option B — 과업(Workstream)을 1급 데이터 레이어로 승격, 7섹션은 그 위 투영
- 장점:
  - 사업의 실제 모양(N과업)과 그릇이 일치 → 학습한 당선 구조가 들어갈 자리 생김.
  - 각 과업이 **RFP 배점 카테고리에 직접 연결**(ADR-006 제1원칙 계승) → 평가위원 채점 축과 출력이 1:1.
  - 키메시지가 과업을 종합, 각 키메시지의 근거를 과업 디테일에서 자동 인용.
  - **확장이 쉬움**: 새 과업유형 = 레지스트리 1행 추가(사장님 "주입 용이" 요구를 구조 수준에서 충족).
- 단점: 데이터 모델 + 생성 파이프라인 + 학습 태깅 동시 변경(범위 큼). 마이그레이션 필요.
- 채택 이유: 목표 3기둥(품질·지속학습·과업구조)을 동시에 푸는 유일한 안. "전부 바꿔도 좋다"는 결정자 승인.

### Option C — 과업유형을 프로파일 메타로만 두고 프롬프트로 처리
- 장점: 코드 변경 거의 없음.
- 단점: 구조가 LLM 프롬프트에만 존재 → 검증·재사용·학습 태깅 불가. 현재 프롬프트 비대화(turn.ts 320줄)의 재발.
- 기각: 구조는 데이터여야 하지 프롬프트면 안 됨(ADR-013 교훈).

---

## Decision

**과업(Workstream)을 1급 레이어로 승격한다.** 프로젝트는 N개의 타입된 과업으로 분해되고, 7섹션 서술과 키메시지는 과업 집합 위의 **투영/합성**이 된다.

### 1. 핵심 모델

```
Project
  └─ Workstream[] (과업 N개)
       ├─ type: WorkstreamType   ← ProgramProfile 6종 확장 (교육·행사_운영·특강연사·장소·모객·심사선발·교류네트워킹·멘토링·산출물 …)
       ├─ scoringCategory        ← 연결 RFP 배점 (ADR-006 제1원칙 계승)
       ├─ detail: Json           ← 타입별 구조화 필드 (레지스트리 스키마)
       ├─ assets: ContentAsset[] ← 이 과업을 뒷받침하는 자산/연사풀/장소DB
       ├─ evidence: Json         ← 당선 근거·수치 (FactCheckLight 대상)
       ├─ budgetSlice            ← 과업별 예산 (전체 예산 = Σ과업)
       └─ supportingPoints: KeyPoint[]  ← 키메시지 뒷받침 디테일
```

### 2. 과업유형 레지스트리 (확장 지점)

`src/lib/workstream/registry.ts` — 타입별로 다음을 한 곳에 선언(asset-registry·channel-preset 패턴 답습):
- **기대 필드 스키마** (예: `특강연사`: 연사후보·주제·섭외난이도·연사료 / `장소`: 후보지·수용인원·접근성·비용 / `행사_운영`: 행사유형·규모·운영인력·안전계획)
- **연결 RFP 배점 카테고리**
- **피드 소스** (어떤 자산/풀이 이 과업을 채우는가)
- **당선 언어 패턴 키** (해당 과업유형의 winning-pattern 태그)

새 과업유형 추가 = 이 레지스트리에 1엔트리. 파이프라인 재작성 불필요.

### 3. 7섹션 = 과업 위 투영

| 섹션 | 합성 방식 |
|---|---|
| ② 추진 전략·방법론 | 과업 집합의 논리 사슬(왜 이 과업 조합인가) |
| ③ 사업 내용 (구 "교육 커리큘럼") | **과업별 블록을 순서대로 렌더** (교육 과업이면 커리큘럼, 행사 과업이면 운영계획) |
| ④ 운영 체계 | 과업별 운영·인력(코치는 멘토링 과업의 한 디테일) |
| ⑤ 예산 | Σ 과업 budgetSlice |
| ⑥ 기대 성과 | 과업별 Output→Outcome(ADR-008 Value Chain) 합성 |

### 4. 키메시지 = 과업 종합 + 과업 근거 인용

- `keyMessages` 캡 3 → 3~5 (사업 규모/과업 수에 비례).
- 각 키메시지의 `messageHierarchy`(이미 존재) 뒷받침 디테일을 **관련 과업의 supportingPoints에서 자동 인용**. "키메시지 + 뒷받침 디테일"이 평문이 아니라 추적 가능한 구조가 됨.

### 5. 학습 루프 정렬 (지속학습 기둥)

- ingest 시 당선 제안서를 **과업유형으로 태깅** (WinningProposalChunk·WinningPattern에 `workstreamType`).
- RAG 검색이 **과업유형별 당선 언어**를 반환 → 생성 시 해당 과업 블록에 배치.
- 결과: 학습량 ↑ = 과업별 출력 품질 ↑ (정렬 회복).

## Consequences

### Positive
- 사업의 실제 모양과 그릇 일치 → 멀티 과업 제안서 품질 구조적 상한 제거.
- 출력이 RFP 배점 카테고리와 1:1 → 평가위원 채점에 직접 대응(당선율 직결).
- 과업유형 확장이 레지스트리 1행 → "주입 용이"를 구조에서 달성.
- 학습 투자(당선 148건)가 출력에 닿음.

### Negative / Trade-offs
- 큰 마이그레이션: Prisma 모델 추가(Workstream/KeyPoint), Express 스키마·생성 파이프라인·UI(S4 워크스페이스) 동시 개편.
- 기존 교육 전용 프로젝트는 "교육 과업 1개" 프로젝트로 자동 변환(하위호환 어댑터 필요).
- 단일 생성 엔진 수렴(3엔진 통합)과 함께 가야 효과 — 별도 ADR-020 필요.

## Teaching Notes (교육자료용)
**신입 PM/개발자가 이 ADR에서 배울 것:**
- 쉬운 길(Option A: 섹션 일반화)은 증상만 가린다. 구조 불일치는 구조로 푼다.
- 구조는 **데이터**여야지 프롬프트면 안 된다(ADR-013 교훈 재확인).
- 이미 있는 자산(ProgramProfile 6종)을 새로 만들지 말고 *승격*하라 — 과업유형은 발명이 아니라 역할 변경.
- 출력 그릇을 RFP 배점에 1:1로 맞추면 "당선 가능"이 평가 가능한 목표가 된다.
- 체크리스트: 새 도메인 요구가 오면 "프롬프트에 넣을까 / 데이터로 모델링할까"를 먼저 묻는다. 검증·학습·재사용이 필요하면 데이터다.
