# ADR-011: Express Mode — "당선 가능한 1차본" 중심 단일 흐름

- 일자: 2026-04-27
- 상태: Accepted (사용자 합의 — 본 ADR 채택과 함께 시스템 정체성 재정의)
- 선행: ADR-001 ~ ADR-010
- 후속: Phase L Wave (L0~L6)

## 결정

UD-Ops Workspace 의 **시스템 정체성을 두 트랙으로 재정의**한다.

```
Express Track (메인)              Deep Track (보조)
────────────────────              ────────────────────
RFP → 30~45분 → 1차본              기존 6 step 파이프라인
챗봇 + Slot Filling                 정밀 산출 (수주 후 실행)
점진 미리보기                      Step 4·5·6 등 디테일
부차 기능 자동 인용 (1줄)           SROI/예산/코치 정밀
                                    Loop 얼라인 Gate
```

- **Express Track 이 메인 진입점**. 신규 프로젝트는 Express 로 시작.
- **Deep Track 은 옵션**. PM 이 1차본 검토 후 "정밀화 필요" 판단 시 진입.
- 기존 6 UI 스텝은 그대로 유지 — Deep Track 의 정체.

**북극성 (North Star)**:
> RFP 한 장 → 30~45분 후 **당선 가능한 기획 1차본 (7 섹션 초안)**

## 배경

### 사용자 통찰 (2026-04-27)

> "언더독스의 강점은 부각이 되지만 RFP에 따라 유연하게 적용 유무를 판단하고 적용하면서, 과정이 가장 사용자 친숙한 방식으로 되려면 어떻게 해야할까?
>
> 복잡도가 올라가는 방식보다는 사용자가 직관적으로 따라가지만, 계속 본인 스스로 흐름을 놓치지 않고 핵심 메세지 중심으로 결과물이 완성되는거야. SROI, 예산, 코치추천 이것도 필요한 기능이지만 부차적이야. **핵심은 RFP에 맞춰서 당선 가능한 기획 1차본이 나오는거지**"

이 한 문단이 시스템 정체성 재정의의 결정적 근거.

### 기존 시스템의 구조적 모순

Phase A~H 누적 결과 시스템이 부차 기능으로 비대해짐:
- SROI Forecast (Phase F) — 정밀 산출 도구, 1차본 단계엔 1줄 인용으로 충분
- 예산 정밀 산출 (Phase A 부터) — 수주 후 실행 도구
- 코치 매칭 (Phase A·E) — 실행 단계 도구
- Loop 얼라인 Gate (Phase F) — 품질 검증, 1차본 후
- ProgramProfile 11축 — 매칭 키, PM 이 일일이 입력하면 부담

**문제**: PM 이 신규 프로젝트 진입 시 "어디부터 시작해야 하는지" 가 불명확. 6 step 모두 차례로 거치면 1차본 도달까지 수 시간 소요. **신입 PM 의 첫 프로젝트 = 막힘**.

### 사용자 기대 산출물

> "RFP에 맞춰서 당선 가능한 기획 1차본"

= 평가위원이 봤을 때 "이거 검토할 가치 있다" 가 되는 7 섹션 초안.
디테일이 아니라 **방향과 차별화**.
시간 = 30~45분 (PM 의 한 timeblock).

## 핵심 설계

### 1. 두 트랙 정체

**Express Track (메인)**
- 단일 화면 — 좌(챗봇 대화) + 우(점진 1차본 미리보기)
- 시간 30~45분, 출력 = 7 섹션 초안 + 키 메시지 3개 + 차별화 5개
- Slot Filling (zod schema) 으로 정형화 보장
- 외부 LLM 분기 (PM 직접 / 외부 LLM / 자동 추출) 카드 패턴
- 부차 기능은 1줄 인용으로 자동 (SROI 추정·예산 마진·코치 카테고리)

**Deep Track (보조)**
- 기존 6 UI 스텝 그대로
- Express 후 "정밀화" 진입점에서 시작
- 수주 후 실행 준비, 평가 대응, 후속 보고 등 깊은 작업

### 2. 1차본 = 7 섹션 (북극성 산출물)

기존 `proposal-ai.ts PROPOSAL_SECTION_SPEC` 7 섹션 그대로:
1. 제안 배경 및 목적
2. 추진 전략 및 방법론
3. 교육 커리큘럼
4. 운영 체계 및 코치진
5. 예산 및 경제성
6. 기대 성과 및 임팩트
7. 수행 역량 및 실적

각 섹션은 **300~600자 초안** (디테일 X, 방향+차별화).

### 3. Slot 정의 (정형화 보장)

zod schema 로 정의된 12 슬롯:

```ts
ExpressDraft {
  // 사업 정체
  intent: string             // 사업의 의도 (1줄)
  beforeAfter: { before, after }  // Before/After 1줄씩
  keyMessages: string[]      // 키 메시지 3개

  // 차별화
  differentiators: Asset[]   // UD 자산 매칭 5개 (자동)
  evidenceRefs: string[]     // 외부 리서치·데이터 근거 (외부 LLM 통해)

  // 7 섹션 초안
  sections: { 1~7: string }  // 각 300~600자
}
```

각 슬롯이 zod schema 통과해야 "다음 단계" 활성화.

### 4. 챗봇 흐름 (Slot Filling + Hybrid)

```
턴 1. AI: "RFP 올려주세요" → PM: [PDF]
       → AI 가 RfpParsed 추출 (자동) → 우측 ① 섹션 일부 채워짐

턴 2. AI: "발주처가 풀고싶은 진짜 문제는?" → PM: 답변
       → AI 가 intent 슬롯 추출 → 우측 키 메시지 1개 표시

턴 3. AI: "이 RFP 와 유사한 사례가 3건 있어요. 활용할까요?"
        → PM: "응" → AI 가 winningPattern 자동 인용 → 우측 ② 섹션 보강

...

턴 N. AI: "1차본 완성! 검토하시고 정밀화 필요하면 Deep 으로."
```

### 5. 외부 LLM 분기 (3 카드 유형 — ADR-007 의 진화)

| 유형 | 동작 | 챗봇이 묻는 방식 |
|---|---|---|
| 📞 PM 직접 | 발주처 통화·메일 (시스템이 못 하는 일) | "발주처에 통화해 보셨어요?" + 체크리스트 자동 생성 |
| 🔍 외부 LLM | 시장·기술·정책 등 외부 자료 | "이건 외부 LLM 에 맡기실까요? 프롬프트 만들어드릴게요" → PM 이 ChatGPT 등에서 답 가져와 붙여넣기 → 챗봇이 슬롯 추출 |
| 🌱 자동 추출 | 시스템이 RFP·자산에서 자동 | PM 에게 알림만 ("자산 X 매칭됨, 기본 활용 OK?") |

### 6. UI 7 장치 (사용자 흐름 놓치지 않기)

1. **북극성 진행 바** (상단 고정) — `RFP 분석 ──● 의도 ──● 차별화 ──○ 1차본` 5~7 단계 진척
2. **핵심 메시지 1줄 카드** (좌측 상단) — "이 사업의 한 문장 요약" 점진 다듬어짐
3. **점진 미리보기** (우측) — 1차본 7 섹션이 ⬜ → 🟦 → ✅ 점차 채워짐
4. **다음 행동 1개** (좌측 하단) — 매 시점 PM 이 할 일 단 1개
5. **부차 기능 접힘** — SROI/예산/코치 정밀은 "심화로 가기" 버튼 뒤. 1차본 흐름 노출 X
6. **Asset 자동 주입** — 매칭 자산을 백그라운드 인용. PM 은 "이 자산이 들어갔어요" 알림 + 확정/제외만
7. **자동 저장** — 매 턴 DB 저장. 이탈 후 재진입 그 자리부터

### 7. RFP 따라 유연한 슬롯 적용

```
필수 슬롯 (RFP 무관)        조건부 슬롯 (RFP 따라)
────────────                ────────────
intent                       SROI 정밀 (평가표 임팩트 ≥20%)
beforeAfter                  지역 맞춤 (region 명시)
keyMessages 3개              방법론 (RFP 가 강조한 IMPACT/UOR)
differentiators 5개          코치 카테고리 (도메인 명시)
sections 1~7
```

ProgramProfile 11축 + RFP evalStrategy 가 자동으로 "이번 RFP 적용 슬롯 vs 생략 슬롯" 결정. **PM 결정 안 봄. 자동 간소화**.

### 8. 부차 기능 자동 인용

| 기능 | 1차본 인용 형태 | Deep 정밀화 |
|---|---|---|
| SROI | "예상 SROI 1:3.2 (벤치마크 기반)" 1줄 | Step 5 의 정밀 SROI Forecast |
| 예산 | "총 예산 N 억, 마진 안전 범위 ✓" 1줄 | Step 4 의 PC/AC 분해 |
| 코치 | "필요 역량 3종 — 매칭 가능 코치 N 명" 1줄 | Step 3 의 코치 배정 |
| 커리큘럼 | "회차 N · IMPACT 단계 매핑" 큰 그림 | Step 2 의 회차별 설계 |

## 비정형 → 정형 안전장치 (사용자 우려에 대한 답)

> "비정형데이터를 정형화 시키는게 굉장히 어려운 로직이 될 것 같아"

4 가지 안전장치:

### A. Schema First
- `src/lib/express/schema.ts` — zod 정의 SSoT
- ExpressDraft, Slot, Validation rules 모두 schema 로 강제
- LLM 출력 = schema 통과해야 인정

### B. Partial Extraction Per Turn
- 사용자 답 들어올 때마다 LLM 이 "이번 답에서 추출한 슬롯" JSON 으로 보여줌
- PM 이 즉시 "맞아 / 수정" 가능
- 누적 실패 방지 (B 옵션 의 단점 회피)

### C. 외부 리서치 분기 자동
- 챗봇이 "이건 모르겠어요" → 자동으로 외부 LLM 프롬프트 생성
- 답 받아 다시 챗봇에 → 챗봇이 슬롯 추출
- 외부 의존성 명확

### D. Validation Gate
- 종료 시 zod 통과해야 1차본 저장
- 미완 슬롯 visible 표시 ("아직 안 채워진 곳 3개")
- 강제 완료 X (PM 이 "지금 1차본으로 이걸로 ok" 결정)

## 대안 비교

### 대안 A (채택): Express + Deep 두 트랙

- 장점: 1차본 30~45분 + 정밀 작업 별도
- 단점: 두 정체 운영, UX 일관성 신경 써야

### 대안 B: 6 step 그대로 압축·다듬기

- 장점: 변화 작음, 학습 곡선 X
- 단점: **사용자 의도 정확히 안 맞음** — "1차본 30~45분" 에 도달 못 함
- 탈락

### 대안 C: Express 만, Deep 폐기

- 장점: 단일 정체, 깔끔
- 단점: 수주 후 정밀 작업 도구 없어짐 (예산·코치 배정 등)
- 탈락

### 대안 D: Free Chat + Final Extract

- 장점: 자유로운 대화
- 단점: 정형화 1회 실패 시 모든 답 유실 — 사용자 우려 정확히 해당
- 탈락 (위 안전장치로 일부 흡수)

## 결과 (기대 효과)

1. **신입 PM 도 첫 프로젝트 30~45분 안에 1차본** — 학습 곡선 급감
2. **부차 기능 노출 X** — 인지 부하↓, 흐름 끊김↓
3. **자동 인용으로 강점 자동 부각** — UD Asset 이 1차본에 자연스럽게 박힘
4. **RFP 별 유연한 슬롯** — 이번 RFP 에 필요한 것만 채워짐
5. **Deep Track 정밀화** — 수주 후 작업 도구 보존
6. **AI 응답 안정성↑** (L1 에서 Gemini + max_tokens + safeParseJson 강화 완료)

## 리스크 + 대응

| 리스크 | 대응 |
|---|---|
| Slot Filling 으로 자유도 부족 → PM "AI 가 묻는 대로만 답해야 하나" 답답함 | 자유 발화 영역 허용 + LLM 이 자유 답에서 슬롯 추출 (Hybrid C 접근) |
| 외부 LLM 분기에서 PM 이 답 안 가져오면 흐름 끊김 | 챗봇이 "외부 답 없어도 진행 가능한 슬롯 먼저" 자동 우선순위 |
| 부차 기능 자동 인용이 부정확 (예: SROI 추정치 잘못) | "이건 추정치 — 정밀화 시 Deep 에서" 명시 라벨 |
| 7 섹션 초안 품질 낮음 (300~600자가 너무 짧음) | 검수 에이전트로 자동 점검 (사용자 요청, Phase L 후속) |
| Express 끝낸 PM 이 Deep 안 가서 정밀화 누락 | 1차본 완성 시 "정밀화 권장 영역 N" 표시 |
