# ADR-005: 가이드북과 ud-ops 시스템의 정체성 분리

**Status:** Accepted
**Date:** 2026-04-16
**Deciders:** 사용자(언더독스), AI 공동기획자
**Scope:** 교육자료 트랙(가이드북) vs 도구 트랙(ud-ops) 의 관계 정의

## Context

2026-04-16 에 사용자가 별도 작업한 `UD Proposal Planning Guidebook v1.0` (HTML) 을 공유함. 메인 세션의 첫 반응은 "가이드북 내용을 시스템에 녹여넣자" 였음 — Phase C 룰 엔진에 가이드북의 사고 프레임을 결정론적 룰로 변환하려는 시도 포함.

사용자의 한 질문("A 와 B 는 맞는 말인 것 같아?") 으로 재검토 → 대부분 부적합으로 판정·철회 ([journey 2026-04-16-guidebook-review](../journey/2026-04-16-guidebook-review.md)).

이후 사용자 명시:
> "가이드북이랑 지금 만드는 시스템은 별개야. 가이드북은 그냥 ojt처럼 배포용이야."

이 관계를 **ADR 로 공식화** 하여 앞으로 유사한 혼동을 예방할 필요.

## Options Considered

### Option A — 가이드북 내용을 시스템에 전면 통합
- 가이드북의 사고 프레임을 AI 프롬프트·룰·UI 힌트로 변환
- 사용자가 읽을 별도 문서 불필요
- **기각:**
  - 가이드북은 **질적 사고법** (평가위원 관점·Underdog 재정의·5대 관점), 시스템은 **결정론적 자동화·정량 검증** — 둘의 역할과 표현이 다름
  - 시스템 안에 가이드북 내용을 담으려면 콘텐츠 CMS 가 됨 — 범위 폭증
  - PM 이 RFP 받기 전·업무 밖에서 학습할 경로가 사라짐

### Option B — 가이드북을 시스템의 부산물로 운용
- 시스템 내 데이터(WinningPattern·ChannelPreset·Journey)를 자동으로 가이드북 형태로 export
- 동기화 자동
- **기각:**
  - 가이드북은 **저자의 주관적 해석·시행착오·교훈** 이 핵심 — 자동화 대상 아님
  - 시스템 데이터는 "무엇이 있는가" 만, 가이드북은 "왜 그래야 하는가" — 자동 변환 불가
  - 자동 export 를 만들면 가이드북 품질이 시스템 데이터 모양에 종속됨

### Option C — 두 트랙 완전 분리 (채택)
- 가이드북은 **OJT 배포용 마크다운 문서**, ud-ops 와 별개 저장소 경로
- 가이드북 내용 → 시스템 반영 **제한적** (브랜드 자산·시드 데이터 경로로만)
- 시스템 변경 → 가이드북 자동 업데이트 ❌, 큰 변경 시 수동 개정 에이전트 호출
- 각 트랙에 **고유 개정 주기** (가이드북 분기 · 시스템 지속 배포)

## Decision

**Option C 채택.** 두 트랙의 역할·저장·개정 절차를 공식 분리.

### 역할 분담

| | 가이드북 | ud-ops 시스템 |
|---|---------|-------------|
| **독자** | 언더독스 PM (신입 포함) | PM 작업 도구 (브라우저) |
| **저장 위치** | `docs/guidebook/` | `src/` + Prisma DB |
| **형식** | 마크다운 (배포 시 PDF/HTML 변환 가능) | 웹 앱 |
| **역할** | 사고 프레임워크 학습 | 작업 실행·자산 축적 |
| **업데이트** | 분기 또는 중대 변경 시 (수동) | 지속 배포 (CI/CD) |
| **담당 에이전트** | `.claude/agent-briefs/guidebook/*` | `.claude/agent-briefs/redesign/*` + 일반 브리프 |

### 정보 흐름 규칙

**가이드북 → 시스템 반영 허용 경로 (제한적):**
1. **브랜드 자산 데이터:** 가이드북에 언급된 수치·자체도구명·Core Values 를 `src/lib/ud-brand.ts` 에 반영 (수치 업데이트만, 해석·톤은 ❌)
2. **시드 데이터 원본 경로 안내:** 가이드북에 기록된 사례의 **원본 PDF** 를 Phase D1 `proposal-ingest` 로 업로드 (가이드북 요약문을 그대로 시드에 쓰는 것은 ADR-003 원본 불변 보존 위반 — 금지)
3. **ChannelPreset 초기 데이터:** 가이드북 Ch.10 발주처 카드 3종 상세 필드를 Phase D2 DB 시드에 이관 (가이드북이 1차 소스, DB 가 2차 캐시)

**시스템 → 가이드북 반영 허용 경로:**
1. **실제 사용 사례:** 시스템으로 수주한 사례는 **수동 개정 에이전트** 가 가이드북 Ch.6~9 에 추가
2. **브랜드 자산 변동:** ud-brand.ts 수치 업데이트 → 가이드북 부록 B 수동 개정
3. **없음:** Journey·ADR·Phase 진행사항은 **가이드북에 반영하지 않음** (내부 문서)

**금지:**
- 가이드북의 사고 프레임 (예: "그래서?" 테스트·Value Chain 인과 검증) 을 Gate 2 결정론적 룰로 변환 — 의미 매칭 불가, false positive 폭증
- 시스템 AI 프롬프트에 가이드북 본문 통째로 주입 — 토큰 낭비·브랜드 보이스는 `ud-brand-voice` SKILL 로 충분
- 가이드북에 시스템 UI 사용법 삽입 — 가이드북은 RFP·PM 사고법 전용

### 운영 주체

- **가이드북 에이전트:** `.claude/agent-briefs/guidebook/` 에 전용 브리프 누적. 최초 작성 (`initial-v2.md`) · 신규 사례 추가 · 브랜드 동기화 · 구조 변경을 각각 별도 브리프로.
- **메인 세션(AI 공동기획자):** 가이드북 에이전트 호출 + 검수 + 커밋. 가이드북 내용을 시스템에 통합 시도 전 반드시 이 ADR 재확인.

## Consequences

### Positive
- 두 트랙 모두 **고유 품질 기준** 유지 (가이드북: 선명한 메시지, 시스템: 결정론적 검증)
- PM 학습 경로와 작업 도구 분리 → 각자 쓰임새가 명확
- 가이드북 사례 추가 비용 낮음 (마크다운 파일 1개 추가)
- 시스템 유지 비용 낮음 (가이드북 동기화 책임 없음)
- 신규 참여자(개발자·PM) 가 어느 트랙을 보면 되는지 명확

### Negative / Trade-offs
- 두 곳에 비슷한 정보 존재 가능 (브랜드 수치 · 발주처 카드)
- 시스템 구조 변경 시 가이드북이 자동 업데이트되지 않음 → 수동 개정 필요
- 사용자가 양쪽을 모두 관리해야 함 (가이드북 독립성 유지 비용)

### Follow-ups
- [ ] 가이드북 v2 최초 작성 (진행 중 — `.claude/agent-briefs/guidebook/initial-v2.md`)
- [ ] 가이드북 분기 개정 체크 포인트 설정 (분기마다 브랜드 수치·수주 사례 업데이트 검토)
- [ ] Phase D2 완료 후: ChannelPreset DB 시드 ↔ 가이드북 Ch.10 정합 확인 에이전트
- [ ] Phase D1 완료 후: WinningPattern Ingestion 결과 중 가이드북 Ch.6~9 추가 후보 선별 에이전트

## Teaching Notes

**신입 PM/개발자가 이 ADR 에서 배울 것:**

- **"좋은 자료가 있으면 어디든 넣자" 는 통합 욕망이 생긴다** — 하지만 역할이 다른 자산은 섞으면 둘 다 망가진다. 가이드북은 읽히기 위한 문서, 시스템은 실행되기 위한 코드.
- **정보 흐름 규칙을 명시한다** — 양방향 전부 열어두면 종속이 생겨 한쪽 변경이 다른 쪽을 깨뜨림. 한쪽에서만 흐르게, 그것도 제한적으로.
- **가이드북은 사람이 읽는 것, 시스템은 기계가 실행하는 것** — 이 경계가 흐려지면 가이드북은 기계적·시스템은 뜬구름이 됨.
- **역할 분리의 비용을 인정한다** — 두 곳에 비슷한 정보, 수동 동기화. 이 비용을 감수하는 게 통합의 "잘못된 단순화" 보다 싸다.
- ADR 체크리스트: 새 자산이 들어오면 ①**독자가 누구?** ②**어느 주기로 개정?** ③**자동화 가치 vs 맥락 보존 가치?** — 이 세 질문이 통합 여부를 결정.
