BRAIN UPGRADE PROPOSAL · v0.2 · 2026-06-12 · ADR-028
Program Design Grammar

프로그램 설계 문법 —
대상이 바뀌면 설계가 어떻게
바뀌는지를 데이터로 만든다

언더독스의 모든 기획안에는 이미 답이 들어 있다. 사전학습을 언제 쓰는지, 온라인과 오프라인을 어떻게 섞는지, 코칭을 몇 회 넣는지, 데모데이를 어디에 두는지 — 대상에 따라 달라져 온 이 암묵지를 명시적 운영 변수 12축과 설계 규칙(Design Rule)으로 구조화하여, 브레인이 커리큘럼·프로그램 설계 1차안을 사람 검수 가능한 근거와 함께 내놓게 한다.

01

진단 — 왜 의도만큼 안 나오는가

자산의 양이 부족한 게 아니다. 수주 패턴 102건, 콘텐츠 자산 1,062건, ProgramProfile 11축, 커리큘럼 검증 룰 4종이 이미 있다. 문제는 "교육 운영의 구조"가 1급 데이터가 아니라는 것이다.

운영 변수가 흩어져 있다

사전학습은 notes 텍스트에, 코호트는 track 문자열에, 성과공유회·데모데이는 formats 라벨에 따로 존재한다. 회차 구조(CurriculumItem)와 연결되지 않아 "몇 회차에 무엇을 어떤 형태로"가 기계가 읽을 수 있는 형태가 아니다.

대상 → 설계 매핑이 암묵적이다

현재 분기는 방법론 스위치(IMPACT·로컬브랜드·재창업 등 9종)와 프롬프트 휴리스틱에 묻혀 있다. "소상공인 대상이면 왜 저녁 시간대 오프라인 격주인가" 같은 근거 규칙이 데이터로 존재하지 않으니, AI가 매번 일반론으로 회귀한다.

학습 단위가 텍스트 조각이다

WinningPattern은 snippet + whyItWorks 서술 단위라 문장 인용에는 강하지만, 제안서 간 커리큘럼·운영 구조를 정량 비교할 수 없다. "수주한 B2G 청년 사업의 평균 코칭 회수" 같은 질문에 지금의 브레인은 답하지 못한다.

03

제안 구조 — 설계 문법의 3층

"대상에 따라 프로그램 설계가 달라진다"를 세 개의 층으로 분해한다. 입력은 이미 있고(Layer A), 어휘를 새로 만들고(Layer B), 문법을 학습한다(Layer C).

LAYER A
입력 축

누구를 위한 사업인가 — ProgramProfile 11축 (재사용)

대상 단계(예비창업·소상공인·임직원·재창업), 인구통계(청년·시니어·상인), 채널(B2G/B2B/연속사업), 규모·기간, 방법론. 새로 만들지 않는다. 이미 모든 프로젝트와 수주 패턴에 박혀 있는 이 11축이 설계 문법의 조건부가 된다.

LAYER B
운영 변수

어떻게 운영하는가 — OperatingFormat 16축 + 콘텐츠 믹스 (v0.2 확장)

사전학습 · 전달 모드 · 회차 리듬 · 회당 시간 · 이론:실습 비율 · 코칭 구조 · 코호트 구조 · 마일스톤 이벤트 · 선발 퍼널 · Action Week · 산출물 · 지원금 · 강사 구성 · 장소 · 수료 기준 · 사후관리, 그리고 VOD 분류 v5.4와 정합하는 콘텐츠 믹스(전달형식 × 콘텐츠유형 × 난이도 곡선)와 회차 시퀀스. 모든 기획안의 교육 운영을 이 좌표로 표현한다. 섹션 04 상세.

LAYER C
설계 규칙

왜 그렇게 설계하는가 — Design Rule (학습 + 검수)

"조건(Layer A) → 권장 운영 변수(Layer B) + 근거 제안서 + 신뢰도" 형태의 규칙. 전체 기획안 재분석에서 통계적으로 도출하고, 수주/패배 결과로 가중하고, 사람이 규칙 단위로 승인해야만 생성에 반영된다. AI는 규칙을 만들 뿐, 무엇이 언더독스의 문법인지는 사람이 결정한다.

기존 Brain 5-Layer와의 관계: Layer B 추출본은 Brain Layer 1(자산)에, Design Rule은 Layer 3(온톨로지)~4(자가강화)에 안착한다. 별도 시스템이 아니라 같은 브레인의 새 차원이다.

04

Layer B — 운영 변수 16축 (OperatingFormat, v0.2 · ADR-028 동결)

04b

VOD 콘텐츠 레이어 — 강의 분류 v5.4 연동 (v0.2 신규)

사용자가 완성한 VOD 학습 콘텐츠 재분류 체계(강의 분류 가이드 v5.4)가 설계 문법의 사전학습·보조학습 슬롯을 구체 콘텐츠까지 내려보내는 해석 공간이 된다. 설계 문법이 "사전학습 = LMS VOD"를 권장하면, VOD 분류가 "이 대상·이 단계에 어떤 영상"을 답한다.

v5.4 분류 축값 공간설계 문법과의 접점

05

대상 × 운영 변수 — 초기 가설 매트릭스

06

재분석 파이프라인 — 기획안 전수 재학습 5단계

학습시킨 모든 제안서를 12축 어휘로 다시 읽는다. 각 단계는 게이트이며, 통과 기준을 만족해야 다음으로 간다.

P1

추출 스키마 동결

OperatingFormat 12축의 키·enum·필수/선택을 확정하고 ADR 후보로 올린다. 여기가 흔들리면 전수 재추출이 무효가 되므로 가장 먼저 닫는다.

게이트: 시드 케이스 10건에 수동 적용해 "표현 못 하는 운영 형태"가 없는지 확인. 산출물: ADR + JSON Schema.

P2

전수 구조화 추출 — 코퍼스 실측 반영 (v0.2)

추출 대상은 WinningProposalDoc 148건 원문(총 186만 자, Drive fetch + OCR 파싱 완료분). AI(plumbing 티어)로 16축 + 콘텐츠 믹스 + 회차 시퀀스를 추출하고, 모든 값에 원문 근거(evidence)와 신뢰도를 기록한다. 저장은 DB가 아니라 JSON-first(data/program-design/) — migration 보류 정책을 지키면서 스키마 이터레이션 자유를 확보하고, 검증 후 DB 이관한다.

게이트: 표본 스팟체크 정확도 90% 이상. ⚠️ 현 코퍼스는 당선작 148 / 패배 0 — 당락 변별 학습은 패배 제안서 수급(사용자) 후 증분 추출.

P3

패턴 마이닝

Layer A 조건(대상 단계 × 인구통계 × 채널 × 규모)별로 12축 분포를 집계한다. "B2G 청년 사업의 코칭 회수 중앙값", "소상공인 사업의 저녁 시간대 비율"처럼 답할 수 있게 만들고, 수주/패배 케이스 간 차이가 있는 축을 표시한다. 경향에서 벗어난 예외 케이스는 버리지 않고 "왜 벗어났는가"를 별도 기록한다 — 예외가 곧 언더독스의 응용력이다.

게이트: 섹션 05 가설 매트릭스의 각 셀에 대해 지지 / 기각 / 데이터 부족 판정 완료.

P4

Design Rule 발행 + 사람 승인

통계가 지지하는 패턴을 "조건 → 권장값 + 근거 제안서 ID + 신뢰도" 규칙으로 발행한다. 규칙은 draft 상태로 태어나고, 검수 UI에서 사람이 근거 원문을 보며 승인·수정·기각한다. approved 규칙만 생성에 반영된다.

게이트: 핵심 세그먼트(청년·소상공인·임직원) 각각 승인 규칙 확보. 산출물: DesignRule 테이블.

P5

생성기 연결 + 설명 가능성

커리큘럼 생성 시 프로젝트의 ProgramProfile에 매칭되는 approved 규칙을 컨텍스트로 주입한다. 생성된 1차안의 모든 운영 결정 옆에 "근거 규칙 + 출처 제안서"를 표시해, 검수자가 결과만이 아니라 이유를 검수하게 한다. PM의 수용/거절은 규칙 신뢰도로 다시 흘러 들어간다(Brain Layer 4 자가강화와 동일 루프).

게이트: 동일 RFP에 대해 규칙 적용 전/후 1차안을 비교하는 블라인드 평가에서 개선 확인.

08

사람 검수 루프 — 어디에서 사람이 개입하는가

시점사람이 보는 것사람이 결정하는 것
P1 스키마12축이 실제 운영 형태를 다 담는가축 추가·삭제·enum 확정 (ADR 승인)
P2 추출표본 추출 결과 vs 원문 대조축별 정확도 합격 판정
P4 규칙규칙 + 근거 원문 + 수주율 차이승인 / 수정 / 기각 — 무엇이 우리 문법인가
P5 생성1차안 + 결정별 근거 규칙 표시설계 수용 / 수정 — 피드백이 규칙 신뢰도로 환류

설계 의도: AI가 패턴을 발견하고, 사람이 문법을 채택하고, 생성은 채택된 문법 안에서만 움직인다. "AI 1차 설계 + 사람 검수"가 결과물 단계에서만이 아니라 규칙 단계에서 한 번 더 일어나므로, 검수 비용이 매 제안서가 아닌 문법에 한 번 든다.
